# Виджеты дашборда: текущий статус и дальнейшие шаги

## Контекст задачи

Цель: привести систему виджетов к предсказуемой модели в стиле Apple:

- фиксированные размеры виджетов;
- понятный выбор размера в каталоге с превью;
- размещение строго в рамках одного экрана (без бесконечного скролла);
- визуальная адаптация содержимого под каждый размер.

На текущем этапе договорились о фиксированных размерах:

- `S = 2x2`
- `M = 4x2`
- `L = 4x4`
- `XL = 8x4`

и о базовой рабочей сетке экрана:

- `16 x 9` (соотношение 16:9).

## Что сейчас работает некорректно (подтвержденные проблемы)

1. **Можно добавить/переместить виджет за пределы видимой области экрана.**
   - Причина: ограничение позиционирования и drop-сценариев в grid пока неполное.
   - Следствие: появляется эффект "экрана со скроллом", хотя это запрещено.

2. **Некоторые виджеты визуально не адаптируются под размер.**
   - Причина: контент внутри ряда компонентов построен как "фиксированный паттерн" без адаптивных режимов по size-классам.
   - Следствие: смена `S/M/L/XL` меняет контейнер, но не всегда меняет структуру внутреннего UI.

3. **Недостаточно выражены продолговатые форматы в реальном контенте.**
   - Причина: шаблоны рендера внутри виджетов не используют отдельные layout-варианты для `M(4x2)` и `XL(8x4)` как "горизонтальные" сценарии.
   - Следствие: визуально мало отличий между конфигурациями, где должны быть явные горизонтальные композиции.

## Что нужно сделать дальше (целевой план)

### A. Жестко зафиксировать экран как одну сцену 16:9

- Ограничить drag/drop так, чтобы элемент нельзя было поставить вне `x:[0..15]`, `y:[0..8]`.
- Любое изменение layout проверять валидатором:
  - если элемент выходит за пределы — откат до последней валидной позиции;
  - если пересечения неразрешимы — откат операции.
- Отключить сценарии, которые увеличивают "виртуальную высоту" рабочей зоны.

### B. Ввести size-aware рендер каждого виджета

Для каждого виджета определить отдельные варианты представления:

- `S (2x2)` — компактная карточка (1 метрика / 1 действие / короткий контент);
- `M (4x2)` — продолговатый горизонтальный вариант;
- `L (4x4)` — базовый расширенный;
- `XL (8x4)` — широкий обзорный.

Это должно быть не просто изменение размеров контейнера, а отдельные композиции контента.

### C. Явно реализовать продолговатые паттерны

- Для `M (4x2)` и `XL (8x4)` использовать горизонтальные сценарии:
  - KPI-лента,
  - горизонтальный список карточек,
  - timeline/полосы статусов,
  - компактные таблицы в одну/две строки.

### D. Каталог: завершить UX

- Всегда отображать кнопку `Добавить` контрастно.
- Блокировать кнопку только если:
  - виджет уже добавлен (если дубли запрещены), или
  - в 16x9 нет места под выбранный размер.
- В превью показывать именно тот вариант UI, который соответствует выбранному size-классу.

### E. Тестовый чек-лист после правок

1. Пользователь не может разместить ни один виджет вне 16x9.
2. После drag/drop не появляется вертикальный скролл рабочего поля.
3. Каждый виджет имеет отличимые `S/M/L/XL` представления.
4. Есть явные продолговатые сценарии в `M` и `XL`.
5. Каталог корректно показывает превью и добавляет выбранный размер.
